## DL-011 — Vova-approved v1 architecture defaults

**Date:** 2026-06-16
**Decision:** Vova approved Hera's recommended v1 defaults:

1. **v1 adapter:** Claude Code only. `auto` resolves to Claude Code. Codex, Gemini, and Antigravity are v2 adapters.
2. **Form factor:** Python CLI first. The CLI may be implemented over internal Python functions; Hermes-native API integration is v2.
3. **ToolRunResult schema:** Python dataclass in code, serialized to YAML in the Run Ledger. JSON Schema is deferred until a second adapter or external consumer needs strict cross-tool validation.
4. **Forbidden Operations:** v1 uses restrictive Claude Code permissions plus post-hoc verification from `tool.log`/`diff.patch`. In-flight hooks are deferred to v2.
5. **Gitignore default:** `.hera/bmad-runs/` is gitignored by default. The `.gitignore` entry is already present in the repo. See DL-006.

**Rationale:** These choices minimize v1 complexity while preserving the core bridge abstraction: Hera orchestrates, Claude Code executes BMAD workflows, Run Ledger captures evidence, and Hera verifies independently. Codex/auth quirks, strict schema design, and tool-specific in-flight hooks are real but should not block the first working loop.
**Captured in PRD:** §5 Non-Goals, §6 MVP Scope, §8 NFRs, §9 Constraints, §10 Dependencies, §11 Open Questions, §12 Assumptions Index.
**Supersedes:** DL-003 uncertainty about v1 adapter scope, DL-005 form-factor uncertainty, DL-006 gitignore assumption, DL-007 forbidden-operation timing uncertainty, and the prior ToolRunResult schema open question.
